# 开发指南

## 读文件顺序

每个新 agent 必须先读：

1. `AGENTS.md`
2. `agent.md`
3. `README.md`
4. `开发指南.md`

## 目录职责

| 路径 | 职责 |
|---|---|
| `enemy-threat-analysis/` | 第一阶段敌情威胁自动分析 Python 包。 |
| `force-grouping/` | 第二阶段作战力量智能编组 Python 包。 |
| `target-allocation/` | 第三阶段作战目标自动分配 Python 包。 |
| `method-planning/` | 第五阶段作战方法/智能路径规划 Python 包。 |
| `terrain_support/` | 共享地形采样模块，按坐标读取 Cesium quantized-mesh 瓦片，不扫描完整 terrain 目录。 |
| `terrain/` | 可选真实地形数据目录，包含 `meta.json / layer.json / {z}/{x}/{y}.terrain`。 |
| `config.py` | 顶层共享配置，现有前两阶段会读取。 |
| `three_part_integration_test.py` | 根目录联调脚本，离线或真实 LLM 串起第 1/2/3/5 部分并输出 JSON、图片、表格、报告和日志；默认检测 `terrain/`。 |
| `run_integration.sh` | 根目录 Bash 联调包装脚本，参数集中在文件顶部，默认使用生成的两份 TXT 输入文档跑真实 LLM 联调。 |

## 共享地形模块

| 文件 | 职责 |
|---|---|
| `terrain_support/__init__.py` | 暴露 `TerrainSampler / TerrainSample / build_terrain_sampler`。 |
| `terrain_support/sampler.py` | 读取 `meta.json / layer.json`，按经纬度定位 `{z}/{x}/{y}.terrain`，解码 quantized-mesh 顶点高程，估算坡度、起伏、地形罚分、速度系数和遮蔽加成。 |
| `tests/test_terrain_support.py` | 使用 synthetic quantized-mesh fixture 覆盖瓦片定位、解码、采样、逐级回退和中性 fallback。 |

## 开发工作流

五部分联调：

```bash
python three_part_integration_test.py --output-dir integration-outputs/latest-three-part-run
```

默认会自动检测根目录 `terrain/`。手动指定地形目录：

```bash
python three_part_integration_test.py \
  --terrain-dir terrain \
  --output-dir integration-outputs/latest-terrain-run
```

联调脚本会持续打印进度，并生成：

```text
integration-outputs/latest-three-part-run/
  json/
  images/
  tables/
  reports/
  run.log
```

第 5 部分产物包括：

```text
json/05-method-planning.json
images/05-route-map.svg
images/05-route-threat-map.svg
images/05-route-terrain-map.svg
images/05-route-combined-map.svg
tables/05-route-table.csv
reports/five-part-integration-report.md
reports/five-part-integration-report.html
reports/five-part-integration-report.docx
```

机降模式联调：

```bash
python three_part_integration_test.py --mission-type air-assault --output-dir integration-outputs/latest-air-assault-method-run
```

由于第四部分纯算法尚未实现，机降模式会自动生成 `json/04-airborne-landing-site-selection.mock.json`，供第五部分验证机降接近/着陆检查点。

固定输出目录重复运行时会默认清理旧产物；加 `--keep-existing-output` 可保留历史文件。

真实大模型联调：

```bash
python three_part_integration_test.py --real-llm --output-dir integration-outputs/latest-real-llm-run
```

真实模式使用第一阶段和第二阶段各自的 LLM 配置。可用 `--skip-assessment` 跳过第一阶段研判 DOCX 调用，用 `--skip-grouping-llm-explanation` 跳过第二阶段推荐解释调用。

第一阶段真实 LLM 输出兼容：

- `enemy-threat-analysis/enemy_threat_analysis/llm_extractor.py` 会在 schema 校验失败后执行 `_coerce_extraction_payload()`。
- 当前兼容模型返回 `targets[].subCategory=null` 的情况，规范为空字符串后保留原字段结构。

根目录也可以直接运行联调包装脚本：

```bash
./run_integration.sh
```

脚本顶部可修改 `REAL_LLM / OUTPUT_DIR / ENEMY_FILES / FORCE_FILES / MISSION_TYPE / TERRAIN_DIR / OBJECTIVE_PREFERENCE / ROUTE_PREFERENCE / ALTITUDE_PROFILE / PHASE_TEMPO` 等变量。默认 `REAL_LLM=1`，输入为：

```text
integration-outputs/generated-input-docs/01_enemy_threat_report.txt
integration-outputs/generated-input-docs/02_blue_force_firepower_report.txt
```

如果只想跑离线 mock，将 `REAL_LLM=0`；脚本会保留其他偏好参数并调用 `three_part_integration_test.py`。

真实 LLM 流式输出：

- `run_integration.sh` 默认导出 `LLM_STREAM=1`、`LLM_TIMEOUT=300`、`ENEMY_THREAT_LLM_TIMEOUT=300`、`FORCE_GROUPING_LLM_TIMEOUT=300`。
- 第一阶段和第二阶段 LLM 抽取器支持 OpenAI-compatible `stream=true` 的 SSE 响应，终端会显示 `[enemy-threat-analysis stream]` 与 `[force-grouping stream]`，并持续打印模型返回的 JSON 片段。
- 流式输出用于提高长文档抽取的超时容忍性和可观测性；它不改变最终 JSON 解析、阶段算法或产物结构。
- 如需关闭终端增量输出，将脚本顶部 `LLM_STREAM=0`；如需保留流式但不打印，可设置环境变量 `LLM_STREAM_STDOUT=0` 后直接调用 Python 脚本。

mobility 处理约定：

- 第二阶段输入输出字段不变，`units[].capabilities.mobility`、`groups[].mobility`、`metrics.mobility` 均表示行进速度，按 km/h 理解，仍限制在 `0-100`。
- 第二阶段真实大模型抽取优先读取文档明示速度；文档缺失时，基于单位类型、装备、角色和世界知识保守推断。
- 第二阶段编组级 `groups[].mobility` 采用慢速约束，取组内最慢单位有效行进速度。
- 第三阶段把 `groups[].mobility` 当速度使用，按距离、速度、预计响应时间、目标波次和能力适配综合分配，不是单纯最近距离分配。

第三部分连续链路约定：

- 同一编组承担多个目标时，语义是连续执行，不是每个目标都从原始阵位重新出发。
- `preferredPlan.assignments[]` 保留既有字段，并新增 `chainId / groupSequence / legIndex / legCount / routeStartCoordinates / routeEndCoordinates / previousTargetId / nextTargetId / chainLegDistanceKm / cumulativeChainDistanceKm / originDistanceKm`。
- `distanceKm` 表示当前实际分腿距离；`originDistanceKm` 保存原始编组位置到目标的审计距离。
- `preferredPlan.taskChains[]` 是第五部分优先读取的链路接口；不同 wave 保持 wave 顺序，同一 wave 内按链路代价选择访问顺序。
- 第三部分目标池会合并四类威胁节点和 `targetAssessments` 真实实体，按 id、来源实体和名称去重；`deploymentSectors` 仍只出现在 `deploymentContexts / targetClusters`。

terrain 处理约定：

- `terrain/` 当前按 Cesium `quantized-mesh-1.0`、EPSG:4326、TMS `{z}/{x}/{y}.terrain` 读取。
- 地形模块只读取 `meta.json / layer.json` 和被请求坐标对应的瓦片；不要写递归扫描 19G 目录的逻辑。
- 第三/第五部分的 `terrain_dir` 是可选参数；地形缺失、越界、瓦片损坏时返回中性结果，不作为硬性 fail。
- 第三部分地形影响分配评分和 reason，但不改变 `distanceKm / reachUtilization` 等公开字段。
- 第五部分地形影响 A* 地形代价地图、路线代价、速度、预计时长和 `[lon,lat,alt]` 高度剖面；`terrain-following` 应使用真实地形高程加安全高度。

第五部分路线生成约定：

- 保留 `analyze -> solve_routes -> build_phases -> build_key_actions -> validate_method_plan` 的任务规划框架，不把时间协同、阶段规划和风险评估塞进 A* 搜索内部。
- `routing.py` 负责把每条 route task 的起点、可选机降地域和目标点转成局部经纬网格，使用 8 邻域 A* 搜索候选路线。
- A* 单元代价由 `threat_cost_at()` 的威胁场、DEM 地形罚分和 `combined_cost_value()` 的偏好权重合成。
- `routes[].plannerMeta` 应保留 `algorithm=a-star-cost-map`、网格大小、展开节点、fallback 段数、搜索边界和 `costLayers.threat/terrain/combined` 摘要，方便演示和排障。
- SVG 路线图默认 `base_layer=combined`；需要演示算法输入时分别渲染 `threat / terrain / combined` 三张底图。
- SVG 路线底图为了演示区分度使用相对色阶：按本图采样分布动态拉伸，并在图例中标注实际色阶范围、峰值和均值。该显示增强不能回写到 A* 搜索代价，规划逻辑仍使用原始威胁/DEM/综合代价值。

第五部分链路路线约定：

- `build_route_tasks()` 优先读取第三阶段 `preferredPlan.taskChains[]`，把一条链拆成多条分腿 route task。
- 第一腿 `routeStartType=group-origin`，后续腿 `routeStartType=previous-objective`，起点为上一目标坐标。
- 跨 wave 不返航；时间上等待下一 wave 窗口，空间上从上一目标继续规划。
- `preferredPlan.routes[]` 保持分腿路线契约，并新增 `chainId / legIndex / legCount / previousObjectiveId / previousObjectiveName / routeStartType`。
- `preferredPlan.routeChains[]` 汇总同一 `chainId` 的路线顺序、总距离、开始/结束时间和目标序列；第六部分仍继续读取 `routes / phases / metrics.totalDistanceKm / missionType`。

部署区处理约定：

- 第一阶段 `deploymentSectors` 表示多个目标聚类出的空间区域。
- 第三阶段输出中，部署区只出现在 `deploymentContexts / targetClusters`。
- `candidateTargets / preferredPlan.assignments / coverage / backlogTargets` 只能包含真实目标实体。
- SVG 图中部署区可以作为半透明背景显示，但分配箭头不能指向部署区。

## 平台对接提醒

- 后期接入平台时，在平台仓库的 `apps/server/src/planning-runtime.js` 注册 `intelligent-allocation`。
- 平台调用时传入 `enemy-threat-analysis` 和 `force-grouping` 的 structuredOutput，再把 `target_allocation.analyze()` 返回值作为第三阶段 structuredOutput。
- 不要删除 `preferredPlan.assignments / coverage / groups / platforms`，后续 `method-planning` 依赖这些字段。
- 如果平台需要地图预览，可在拿到 structuredOutput 后调用 `target_allocation.visualization.render_allocation_map()` 生成 SVG。
- 后期接入第五部分时，在平台仓库注册 `intelligent-planning`，并把 `enemy-threat-analysis`、`target-allocation`、可选 `force-grouping` 和可选 `airborne-landing-site-selection` structuredOutput 传给 `method_planning.analyze()`。
- 不要删除第五部分 `preferredPlan.routes / phases / metrics.totalDistanceKm / missionType`，第六部分保障规划依赖这些字段。
